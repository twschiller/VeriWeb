What are preconditions and postconditions?

A precondition is an expression that must be true when a method is called. For example, the following method has the pre-condition x != null:

 
String convertToString(Integer x){ return x.toString(); }

because if x is null, x.toString() will cause a NullPointerException.

A postconditions is an expression that describes what must be true when a method returns. For example, the following method has the postcondition \result == (x == 5):

 
boolean isFive(int x){ return x == 5; }

because the method returns whether or not x equals 5.

NOTE: Preconditions and postconditions can only refer to method parameters and object fields (member variables).

I can't satisfy one of the method's preconditions, what should I do?

Either (1) the condition you need isn't listed or (2) the precondition is bad:

  1. A precondition you need for the current method isn't listed: take a look at our answer to the question Where is the condition I need?
  2. The precondition is bad, view the documentation for the corresponding method and click the "Mark as Bad" button next to the pre-condition. We'll ask you to describe the bad condition so that we can tell someone (maybe even you) to remove it.

Why doesn't the system know anything about the methods that are called by this method?

The system can't always guess conditions automatically, so we need people like you add them for us. To view what the system knows about a method's conditions, view the method's documentation by hovering your mouse over the method name. You can also toggle the inline documentation for some methods by clicking on the method name — this will show both facts that that system hasn't been able to prove yet and facts that are known to be false.

It could be that the system doesn't know that calling the method preserves / establishes the property you're interested in. To tell it, view the documentation for the method and click the "Add" button next to the list of preconditions; we'll ask you to describe the condition so that we can tell someone (maybe even you) to add it.

Why does the system think that a method that is called by this method will throw an exception?

For methods that throw exceptions, the system may not fully understand under what circumstances an exception is thrown. To tell it, view the documentation for the method and click the "Add" button next to the exceptional postconditions; we'll ask you to describe the condition so that we can tell someone (maybe even you) to add it.

What are these messages from other methods?

A user working on a different problem thought that this method was either missing a condition or that one of the conditions was wrong. The messages explain which conditions they thought were missing or incorrect.

What are the colored boxes that contain conditions?

The conditions you submit are displayed here; we'll sometimes try out some conditions automatically for you, too.

Conditions in green have been accepted by the system, whereas conditions in red have been rejected by the system. A blue information icon "i" will appear next to a condition when we can tell you why the condition was rejected. Hover your mouse over the information icon to see the reason.
